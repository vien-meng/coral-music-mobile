## 目标、范围、不做内容与依赖

当当前曲目的解析、加载或播放发生错误时，在本轮队列中自动尝试下一首未失败歌曲；所有可选歌曲均失败后保留明确错误状态，绝不无限循环重试。

依赖 B4-01、B4-03、B4-04、B4-05。不实现换源、音质降级、网络指数退避、失败持久化或错误上报；它们需要来源/缓存和诊断能力后另行实施。

## 桌面端基线与确认行为

对照桌面端 `coral-music-desktop/src/renderer-react/stores/domains/playerStore.ts` 的统一队列切换路径。桌面端播放失败不能让列表播放永远停在同一失效项；移动端沿用既有队列选歌和 `playTrack()` 入口，不在 UI 侧重复错误策略。

## 实施方案、关键接口、数据与平台差异

- `PlayerController` 在当前曲目失败时记录本轮失败的稳定 `Track.id`，然后按队列当前模式取得下一项。
- 只自动尝试尚未失败的不同歌曲；单曲或全队列均失败则保留 `AppFailure`，以避免循环请求失效 URL。
- 用户手动选择歌曲时允许重新尝试该项；不改平台音频、User API 和数据库。

## 验收、风险与恢复入口

- 收口时覆盖“第一首失败跳至第二首”“两首均失败后停止”“手动重试已失败歌曲”三类单元测试，并在 Android 真机用受控失效 URL 加有效 URL 回归。
- 当前恢复入口为 `lib/features/player/state/player_controller.dart`；实际改动和验证将在任务结束时补充。

关联 P3-03、功能矩阵“生命周期”“队列”；后续为音质降级/换源和播放进度持久化。

## 当日实施进度

- 无可用候选时保留明确错误状态，手动点歌会移除该曲失败标记并允许重新尝试。受控失效 URL 真机回归待后续集中执行。

## 2026-07-17 失败策略回归

- 新增 `skips a track whose URL cannot be resolved`：首曲 User API 解析失败后，队列选择第二曲并进入真实控制器的播放态。
- 新增 `keeps an error when every queued track fails`：单曲队列失败后保留“测试取链失败”，没有再次请求相同失效曲目。
- `flutter test test/player_controller_test.dart -r compact` 通过 12 项、相关 Dart 分析通过。Android 受控失效 URL 与有效 URL 的真机回归仍待设备重新连接，任务保持 `DOING`。

## 2026-07-17 最终失败恢复入口（DOING）

- 审计发现自动跳过穷尽后，播放详情和迷你播放栏只显示错误文本；虽然再次点播放按钮会调用 `playTrack()`，但缺少清晰的恢复意图，用户难以判断是否会重新取链或仅切换暂停。
- 将在播放详情的最终错误状态显示“重试播放”，直接调用现有 `playTrack(track)`；该路径会清除该曲失败标记、允许 URL 刷新和质量降级，不新增 UI 层重试循环或网络逻辑。

## 2026-07-17 最终失败恢复入口（DONE for shared UI）

- 播放详情在 `player.error` 存在时显示“重试播放”按钮；它调用已有的 `PlayerController.playTrack(track)`，没有复制 URL 刷新、音质降级或自动跳过策略。
- `playTrack()` 默认 `retryFailed: true`，因此手动重试会删除该曲本轮失败标记并允许完整解析链重新运行；原有控制器单测继续覆盖“所有歌曲失败后保留错误”和“失败后跳过下一首”。
- 验证：`dart format lib/features/player/view/player_detail_page.dart`、`flutter analyze --no-fatal-infos` 和 `git diff --check` 通过。真实 LX 音源当前可正常取链，未人为引入无效地址；受控失败的 Android 真机回归仍保留为 B4-06 后续验收项。
