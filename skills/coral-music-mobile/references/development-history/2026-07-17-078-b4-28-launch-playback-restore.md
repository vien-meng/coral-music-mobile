## 目标、范围、依赖与不做内容

启动应用后从播放历史恢复最近一首曲目的封面、名称和可继续播放位置，使迷你播放器与播放详情不再显示空状态。依赖 B4-16 的历史记录与 `LibraryStore.listHistory()`。

不做内容：应用启动自动播放、自动重新解析在线 URL、恢复完整队列、恢复动态 User API 脚本、跨设备同步。在线 URL 与脚本都可能失效；用户明确点击播放时才走既有 `playTrack()` 解析链。

## 桌面端对照与行为

- 桌面播放器保存最近播放上下文；移动端采用平台更安全的等价行为：恢复为未加载的暂停曲目，而不是后台/启动时擅自联网播放。
- 最近位置少于 5 秒、或已经接近曲目结尾时按零位置恢复，与既有历史“继续播放”阈值保持一致。

## 实施方案与验收

- 在 `PlayerController` 增加幂等的 `restoreLastPlayback()`：仅当当前没有活动曲目时读取历史第一项，设置 `idle` 状态与合法的保存位置，不触碰音频引擎。
- 在 `ProviderScope` 内的应用启动包装组件调用一次；数据库不可用时静默保留空播放器，避免启动被历史功能阻塞。
- 验收将覆盖：有历史恢复、短位置归零、已有正在播放状态不被覆盖，以及首次点击播放仍会通过 User API 重新取链。

## 2026-07-17 实施与验证

- `PlayerController.restoreLastPlayback()` 已通过既有 `LibraryStore.listHistory()` 恢复第一条历史的 `Track`、合法进度、默认质量、倍速与音量，状态保持 `idle`，不会触碰 `AudioEngine` 或在启动时发起 URL 解析。
- `toggle()` 对当前恢复的曲目会传回保存的位置，用户首次播放后才走 `PlaybackResolver` 并 seek；低于 5 秒或接近曲尾的位置归零，沿用 B4-16 的恢复规则。
- `CoralMusicApp` 在 `ProviderScope` 内只调度一次恢复；数据库错误被视为可选启动上下文失败，不阻断路由与首页。
- 已补充控制器用例，但按当前“先完成业务开发”的节奏未单独执行。统一验证已通过：`dart format`、`flutter analyze --no-fatal-infos`、`flutter build ios --no-codesign` 与 `git diff --check`。Android/iOS/Harmony 重启真机验收仍待后续集中进行。

## 当前状态与恢复入口

- 状态保持 `DOING`，原因是三端冷启动、在线源失效和本地/下载/WebDAV 四类来源的真机行为尚未集中验收。
- 恢复入口：杀掉应用后重新启动；迷你播放器应显示最近历史曲目。点击播放才重新取链并从保存位置继续。

## 2026-07-17 在线恢复的音源会话提示

- 在线曲目恢复后，动态 User API 会话按安全设计不会跨重启保留。`MethodChannelUserApiRunner` 现在在没有已启用 manifest 时明确提示“请先在音源管理导入并启用音源”，不再误报为泛化的“当前音源不支持”。
- 这只改善恢复入口的可理解性；不会自动恢复脚本、不绕过会话隔离，也不会在启动时联网。
- 播放详情在该明确错误下提供“去导入音源”直达入口，保留“重试播放”用于已经重新导入后的同曲继续；不为其它网络/脚本错误显示误导性的设置跳转。
