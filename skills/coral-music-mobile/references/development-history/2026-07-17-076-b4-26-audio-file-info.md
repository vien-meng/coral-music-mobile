## 目标与范围

在播放详情显示桌面端同口径的音频文件信息：实际探测到的 `kbps · kHz · 容器格式 · 音质档位`；探测失败时回退到桌面端已有的质量规格映射。

不做：下载完整音频、写入媒体标签、音频转码或把质量档位文本当成实际探测值。

## 桌面端对照与方案

- `coral-music-desktop/src/renderer-react/services/playerRuntime/audioHeaderProbe.ts`：使用 HTTP Range 读取前 64KB，解析 MP3/FLAC 文件头。
- `coral-music-desktop/src/renderer-react/stores/domains/playerStore.ts` 的 `bitrateText`：优先真实探测；未命中时用质量规格给出明确回退。
- 移动端将以同样的 64KB Range 请求和 MP3/FLAC 头解析实现，不引入依赖；结果只绑定当前播放请求，切歌后的旧探测不会覆盖新曲目。

## 验收

- FLAC 文件头解析出采样率、估算 PCM 比特率和 FLAC 格式；MP3 解析出帧头码率与采样率。
- 真机播放详情显示类似 `1323 kbps · 44 kHz · FLAC · SQ`，探测失败时按已选择档位回退。

## 2026-07-17 FLAC 实际平均码率修订（DOING）

- 用户以桌面端信息条为基线，要求显示真实文件比特率。现有 FLAC 头探测只用采样率、位深和声道数计算 PCM 理论码率，无法反映 FLAC 压缩率，因而可能显示 `1411`/`2117 kbps` 而非文件实际平均值。
- 移动端将只在同一 Range 响应含 `Content-Range` 总文件大小（或服务端完整 `Content-Length`）且 FLAC STREAMINFO 含总采样数时计算 `totalBytes × 8 × sampleRate / totalSamples`；缺任一数据时保留桌面兼容的 PCM 估算回退。
- 该改动不额外下载文件、不改变播放请求或文件格式解析边界；将用纯头部样本覆盖实际平均码率和回退两条路径，再以真实 LX FLAC 播放回归。

## 2026-07-17 移除未探测参数的规格伪回退（DONE for shared UI）

- 用户明确要求文件信息是实际音频参数。此前 UI 在探测还未返回或失败时会按 SQ/HQ 等枚举拼出理论 `kbps · kHz`，而 FLAC 在缺少完整文件长度时还会用 PCM 理论码率替代压缩流平均码率；两者都可能被误解为真实文件信息。
- 现改为：仅显示探测到的比特率、采样率和容器格式；未取得这些字段时只显示实际播放质量标签。FLAC 只有同时取得总文件字节数和总采样数时才提供平均码率，绝不再显示 PCM 理论值。
- 验证：`does not invent a FLAC bitrate without complete file metadata`、真实总大小平均码率及 MP3 帧头共 3 项在 `flutter test test/audio_file_probe_test.dart -r compact` 通过；`flutter analyze --no-fatal-infos`、`git diff --check` 通过。已取得的真机 `1643 kbps · 44 kHz · FLAC · SQ` 证据不受影响。

## 2026-07-17 MP3 ID3 长度偏移修正

- 审计发现 ID3v2 标签的跳过位置此前用按位或计算 `10 | size`，而正确语义是固定 10 字节头加上 syncsafe 标签长度；特定标签长度会导致扫描不从第一个 MP3 帧开始，进而缺失真实 `kbps/kHz`。
- 已改为 `10 + size`，保持已有 64 KiB Range、帧头扫描和无探测值不伪造规格的边界不变。
- 本轮按“先完成业务开发”节奏仅执行格式与差异检查；MP3 带 ID3 边界样本会在下一次聚合播放器回归补跑。
